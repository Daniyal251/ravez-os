# RAVEZ OS — План реализации для владельца

## Зачем этот документ

Этот файл для тебя как для владельца продукта (без необходимости кодить).
Здесь зафиксировано:
- что уже сделано,
- что не работает и почему,
- какой итоговый план согласован,
- как мы реализуем всё по порядку.

## 1) Что у нас есть сейчас

- Новый фронт на `React + Vite + Tailwind` уже создан.
- Базовые страницы есть: `Login`, `Dashboard`, `Tasks`, `Events`, `Team`, `Shifts`, `Finances`.
- Есть техТЗ: `docs/TECH_SPEC_RAVEZ_OS.md`.
- Добавлен тестовый локальный вход (`demo/demo`) через `.env.local` и mock API.
- Подключена папка наследия: `старое/staff.html`, `старое/staff-app.js`.

## 2) Главная проблема (честно)

Проект исторически собирался разными ИИ по частям, поэтому:
- структура местами не единая,
- UX непоследовательный,
- часть логики дублируется,
- есть модули, которые были в старой системе, но еще не перенесены в новую.

Это нормальная стадия миграции. Решение: не "чинить хаос", а последовательно собирать целевую систему по этапам.

## 3) Уже согласованная целевая модель

Мы уже зафиксировали:
- оргструктуру: ГД -> ЗГД -> РП -> Модули (+ ветка ЗГД -> Модуль напрямую),
- модульную командную схему,
- бюджетирование по направлениям (план/факт, лимиты, согласование),
- RBAC (роли и права),
- приоритет миграции старых модулей в новую архитектуру.

Подробности: `docs/TECH_SPEC_RAVEZ_OS.md`.

## 4) Что берем из старой системы (обязательно)

Из `старое/staff-app.js` переносим в новую версию без потерь:
- роли и матрицу прав;
- задачи, смены, финансы;
- зарплаты (настройки, начисления, выплаты);
- CRM гостей и лояльность;
- интеграции и рассылки;
- аналитические панели.

## 5) Порядок реализации (фиксированный)

### Этап P0 (ядро, без этого дальше нельзя)
1. Стабильный Auth и навигация.
2. Роли и базовый доступ в UI.
3. Team, Tasks, Shifts, Finances — рабочие end-to-end.
4. Единые состояния loading/error/empty/save.

### Этап P1 (операционное расширение)
1. Salary (настройки, расчет, начисления, выплаты).
2. Guests CRM.
3. Loyalty (тиеры, правила, начисление/списание).

### Этап P2 (масштабирование)
1. Integrations.
2. Broadcast/сообщения.
3. Продвинутые панели (Owner/Thursday/Party/Content/Promoters).
4. Полировка UX и аналитики для руководства.

## 6) Как работаем, если ты не кодишь

Твоя роль:
- утверждать бизнес-правила,
- приоритизировать задачи,
- принимать готовые экраны по чек-листу.

Моя роль:
- делать техреализацию по этапам,
- объяснять простыми словами,
- проверять, чтобы ничего не ломалось при каждом шаге,
- вести единый порядок, чтобы не было "хаотичных патчей".

## 7) Что считаем "готово"

Система считается готовой, когда:
- каждый ключевой модуль работает от входа до сохранения;
- роли ограничивают доступ по правилам;
- бюджетирование работает по направлениям;
- аналитика показывает план/факт и контроль исполнения;
- мобильный и десктоп UX одинаково удобны.

## 8) Следующий практический шаг

Начинаем с P0 и делаем короткими итерациями:
1) фикс навигации и role-gating,
2) унификация API-ошибок,
3) доведение Team/Tasks/Shifts/Finances до рабочего "боевого" состояния.

Каждую итерацию закрываем форматом:
- что сделано,
- что проверено,
- что осталось в этом модуле.
